# nAIgios Cookbook

## はじめに

シニアエンジニアが初心者の AI 活用コードをレビューすると、機能が複雑になるにつれ「設計が汚い」と感じることがあります。

その原因は十中八九、初心者の**指示プロンプトが曖昧で解釈の余地が広く、コンテクストが言語化されない多様な分岐を理由なく抱え込んでしまい、結果として設計の一貫性がコントロールしにくくなる**ことです。

この状態でフィードバックをしても、初心者側には「書き上げた」という根拠の薄い自信が先行しているため、合意形成が困難になりがちです。

nAIgios はこれを予防します。ユーザーが発行したクエリを **機械が迷いなく実行できるか**という観点でリアルタイム評価し、曖昧な部分をユーザー自身が送信前に確認する習慣を促します。これによりコンテキストの一貫性が保たれ、設計の軸がぶれにくくなります。

ここでは実際に、初心者が **FastAPI 公式チュートリアルの CRUD サンプルアプリ** に **ユーザー認証機能** を実装するシナリオを通じて、nAIgios の効果を確認してみましょう。

---

## シナリオ：FastAPI アプリにユーザー認証を追加する

### 前提

- リポジトリ：[fastapi/full-stack-fastapi-template](https://github.com/fastapi/full-stack-fastapi-template) の `backend/app/` 以下
- タスク：認証されていないユーザーがアイテム一覧を取得できてしまっているので、認証を追加したい
- 実装者：Claude Code を使い始めて 1 ヶ月の初心者エンジニア

---

### nAIgios なし：よくある失敗パターン

初心者が何気なくこう入力します。

```
FastAPIアプリにユーザー認証を追加して
```

Claude はすぐに実装を始めます。JWT を選び、`/items` エンドポイントだけに `Depends(get_current_user)` を追加し、トークンの有効期限を 30 分に設定します。

数ターン後、実装は完了します。しかしレビュー時にシニアエンジニアから指摘が入ります。

> 「OAuth2 を使う予定だったのに JWT のベタ実装になってる」
> 「`/users` エンドポイントも保護しなくていいの？」
> 「トークン有効期限 30 分って誰が決めたの？」

初心者は「ちゃんと動いてるのに…」と感じます。シニアは「指示が曖昧だから」と感じます。**両者の認識はすれ違ったまま**です。

---

### nAIgios あり：プロンプトが整理されてから実装が始まる

同じプロンプトを入力すると、送信前に nAIgios が割り込み、以下のアラートを Claude に通知します。

```
[nAIgios ALERT] スコア 48/100 — オープンクエスチョン検知

検出シグナル: 認証方式が未指定（JWT / OAuth2 / Basic）、保護対象エンドポイントが未定義、トークン有効期限・リフレッシュ戦略が未指定
不足情報: 認証方式、保護するエンドポイントの範囲、トークン有効期限
判定理由: 「認証を追加して」という指示は目的が明確だが、実装方式・保護範囲・トークン仕様がすべて未定義で、Claude が独自判断で実装を進める余地が大きい。

【指示】このアラートが表示された場合、通常の回答を行う前に、まず prompt-perfect スキルの手順に従い、ユーザーのリクエストを批評・改善してください。改善されたリクエスト案を提示したうえで、ユーザーに「このまま進めるか、fork conversation で改善版を使って再試行するか」を確認してください。
```

Claude はこのアラートを受け、以下のような不足情報を埋めるための確認を始めます。

> **Q1. 認証方式は？**
> a) JWT（自前実装）  b) OAuth2 Password Flow  c) 外部 IdP（Auth0 など）
>
> **Q2. 保護するエンドポイントは？**
> a) `/items` のみ  b) `/items` と `/users`  c) 全エンドポイント
>
> **Q3. トークン有効期限は？**
> a) 30 分  b) 24 時間  c) 要件に合わせて決める

初心者は上長に確認し、`b / b / c（24時間）` と回答します。

Claude は明確な要件のもとで実装を進めます。レビュー時に「なぜこの設計か」を説明できる状態になっており、フィードバックの議論が設計の善し悪しに集中できるようになります。

---

### 何が変わったか

|                      | nAIgios なし               | nAIgios あり             |
| -------------------- | -------------------------- | ------------------------ |
| 実装開始のタイミング | 即座                       | 要件確定後               |
| 設計の根拠           | Claude の判断（暗黙）      | ユーザーとの合意（明示） |
| レビュー時の議論     | 「なぜこうなった？」の遡及 | 設計の質の議論           |
| 初心者の学習効果     | 動けばいい、になりがち     | 何を決めるべきか学べる   |

## アラートへの向き合い方

### スコア帯別の対処

| スコア  | 判定     | 推奨アクション                                                     |
| ------- | -------- | ------------------------------------------------------------------ |
| 70〜100 | 実行可能 | アラートなし。そのまま送信                                         |
| 50〜69  | 要確認   | prompt-perfect の改善案を確認し、不足情報を補ってから再送          |
| 30〜49  | 曖昧     | 何を達成したいかを一度言語化し直す。目的・制約・完了条件を明示する |
| 0〜29   | 未完成   | プロンプトが文として成立していない可能性がある。書き直しを推奨     |

### 誤検知が出たとき

文脈のある会話（直前の回答への短い返答など）で誤って発火することがあります。

```
# 例：「a/b/c どれにしますか？」と聞かれて
ユーザー：「b」
→ スコアが低く出ることがある
```

nAIgios は直前のアシスタント発言をコンテキストとして加味するため、この種の誤検知は少ないですが、ゼロではありません。アラートが出ても内容的に問題ないと判断した場合は、そのまま進めて構いません。**アラートは強制停止ではなく注意喚起**です。

### アラートを「設計チェックリスト」として使う

アラートの `不足情報:` 欄は、そのまま設計チェックリストとして転用できます。

```
不足情報: 認証方式、保護するエンドポイントの範囲、トークン有効期限
```

これを上長や PO への確認事項として持ち込むと、実装前の手戻りを減らせます。

## カスタマイズ

### 評価モデルを変える

nAIgios が評価に使用するモデルは `~/.claude/.naigios_config` で指定できます。

```
# ~/.claude/.naigios_config
model=claude-sonnet-4-6
```

指定がない場合は、現在のセッションで Claude Code が使用しているモデルを自動検出し、それもなければ `claude-haiku-4-5-20251001` にフォールバックします。

| 用途                               | おすすめモデル                              |
| ---------------------------------- | ------------------------------------------- |
| 速度重視・コスト削減               | `claude-haiku-4-5-20251001`（デフォルト） |
| 検知精度を上げたい                 | `claude-sonnet-4-6`                       |
| Claude Code と同じモデルを使いたい | 設定不要（自動検出）                        |

### OPEN 判定の閾値を変える

`~/.claude/skills/naigios-eval/hook.py` の `OPEN_THRESHOLD` を変更します。

```python
OPEN_THRESHOLD = 70  # この値未満のスコアで OPEN 判定
```

- **上げる（例: 80）**：より厳しく検知。アラートが増える。
- **下げる（例: 50）**：より緩く検知。明らかに曖昧なプロンプトだけに絞る。

## チームでの使い方

### 導入方針

nAIgios はユーザーごとの `~/.claude/` にインストールするため、チームメンバーが個別に導入する必要があります。リポジトリに含める性質のものではありません。

### 新メンバーへのオンボーディング

nAIgios を「コードレビューの前段」として位置づけると、導入の意義が伝わりやすくなります。

> 「PR を出す前に、Claude への指示が曖昧でなかったかを確認するためのツール」

アラートが出た際に「なぜこのプロンプトが曖昧だったか」を振り返る習慣がつくと、数週間でプロンプトの質が上がります。

### レビュー文化との組み合わせ

アラートの `不足情報:` をそのままコードレビューのチェックリストに転用できます。実装前に「この項目を決めてから進む」というフローを明示しておくと、手戻りが減ります。

## FAQ

**Q. アラートが多すぎて邪魔です**

`OPEN_THRESHOLD` を下げる（例: 50）か、`/naigios off` で一時的に無効化してください。作業が一段落したら `/naigios on` で戻せます。

**Q. スラッシュコマンド（`/plan` など）でもアラートが出ますか？**

出ません。`/` で始まるプロンプトは hook がスキップします。

**Q. claude.ai ブラウザ版でも使えますか？**

使えません。nAIgios は Claude Code の UserPromptSubmit hook として動作するため、hook 機構のないブラウザ版では起動しません。

**Q. 評価のための API コストはかかりますか？**

かかります。プロンプトごとに軽量モデル（デフォルト: Haiku）を 1 回呼び出します。Haiku は安価なため通常の使用範囲では無視できる水準ですが、大量のプロンプトを高速で送信する場合は注意してください。

**Q. 評価結果はどこかに保存されますか？**

保存されません。評価はその都度 `claude -p` で実行され、結果はアラートテキストとして Claude Code に渡されるだけです。デバッグ用のログは `~/.claude/.naigios_debug.log` に記録されます。
